Sensors & Actuators — IO Definition

RIVO Node uses declarative configuration files to define hardware IO. No recompile is needed — simply add or edit a file and restart the node. All defined capabilities are automatically reported to the Server and made available to the AI assistant.

Two file types:
  • Sensor Protocols — YAML files in protocols/, define how to read raw data from UART, I2C, SPI, Modbus, GPIO, 1-Wire, CAN.
  • Actuator Interfaces — JSON files in interfaces/, define controllable outputs and expose them to the AI assistant and command API.

1. Sensor Protocol YAML

Each file in protocols/ describes one sensor or device. The filename is arbitrary; the id: field is what links it to the UART/I2C configuration in config.yaml.

Top-level fields

Field Required Description
idYesUnique protocol ID — referenced in config.yaml uarts.<name>.protocol
typeNoSensor type label (free text, e.g. particulate_matter)
busYesuart / i2c / spi / modbus / gpio / onewire / can
addressI2C only7-bit I2C address in hex, e.g. "0x76"
capabilitiesYesList of named data outputs (see below)
readYesRead timing and field decode rules
mapNoMaps decoded fields to telemetry topics and status fields
queryNoCommand-response pattern (send a command, then read response)
initNoOne-time register writes at startup (I2C/SPI device init)
identifyNoWHO_AM_I verification before starting reads

capabilities[ ] — Output Definitions

Each capability describes one data output from the sensor:

capabilities:
  - name: "pm25"                    # unique name within this protocol
    category: "sensor"              # "sensor" | "state" | "actuator"
    data_type: "float"              # "float" | "int" | "bool" | "string"
    unit: "ug/m3"
    range: [0, 999]                 # [min, max] — optional, used by AI safety checks
    telemetry_topic: "/air_quality" # topic pushed to server telemetry
    status_field: "pm25"            # key in robot status JSON
    sample_interval_ms: 1000
    description: "PM2.5 particulate matter concentration"

read — Field Decode (UART / binary)

read:
  interval_ms: 1000   # polling interval
  fields:
    - name: "pm25"
      offset: 6        # byte offset in the frame
      len: 2           # byte length
      decode: "uint16_be"  # uint8 | uint16_be | uint16_le | int16_be | float32_be | ...
      scale: 0.1       # optional multiplier after decode
      offset2: 0.0     # optional additive offset after scale

read — Register Read (I2C / SPI)

read:
  interval_ms: 500
  registers:
    - name: "temperature"
      reg: "0xFA"      # register address
      len: 3           # bytes to read
      decode: "int20_be"  # see decode types table

Supported decode types

Decode Type Bytes Description
uint81Unsigned 8-bit integer
uint16_be / uint16_le2Unsigned 16-bit big/little endian
int16_be / int16_le2Signed 16-bit
uint32_be / uint32_le4Unsigned 32-bit
float32_be4IEEE 754 float, big endian
int20_be3Signed 20-bit (BME280 raw ADC)
bcd22BCD encoded value

map — Telemetry & Status Mapping

map:
  status:
    sensor_overview:
      - sensor_id: "PM2.5"
        sensor_type: "Particulate Matter"
        value_field: "pm25"
        unit: "ug/m3"
  telemetry:
    "/air_quality":
      pm25:
        field: "pm25"
        unit: "ug/m3"
      pm10:
        field: "pm10"
        unit: "ug/m3"

query — Command-Response Sensors

For sensors that require a trigger command before each read (e.g. SDS011):

query:
  enabled: true
  command: [0xAA, 0xB4, 0x04]   # UART: bytes to send
  wait_ms: 200                   # wait before reading response
  response_length: 10            # expected response byte count
  verify:
    header: [0xAA]
    checksum: "sum_complement"   # "sum_complement" | "crc8" | "crc16_modbus" | "none"

Modbus RTU sensor

bus: "modbus"
modbus:
  slave_id: 1
  function_code: 4    # 3=holding registers, 4=input registers
  start_reg: 0
  count: 4
  registers:
    - name: "temperature"
      index: 0        # 0-based register index in response
      decode: "int16"
      scale: 0.1
    - name: "humidity"
      index: 1
      decode: "uint16"
      scale: 0.1

GPIO sensor (HC-SR04 ultrasonic)

bus: "gpio"
gpio:
  mode: "echo_distance"
  trigger_pin: "GPIO17"
  echo_pin: "GPIO18"

capabilities:
  - name: "distance_cm"
    category: "sensor"
    unit: "cm"
    range: [2, 400]
    telemetry_topic: "/distance"

Complete UART example — SEN0177 PM Sensor

id: "uart-sen0177"
type: "particulate_matter"
bus: "uart"
uart:
  baud: 9600
  framing: "fixed"
  length: 32
  header: [0x42, 0x4D]

capabilities:
  - name: "pm25"
    category: "sensor"
    data_type: "float"
    unit: "ug/m3"
    range: [0, 999]
    telemetry_topic: "/air_quality"
    status_field: "pm25"
  - name: "pm10"
    category: "sensor"
    data_type: "float"
    unit: "ug/m3"
    range: [0, 999]
    telemetry_topic: "/air_quality"
    status_field: "pm10"

read:
  interval_ms: 1000
  fields:
    - name: "pm25"
      offset: 6
      len: 2
      decode: "uint16_be"
    - name: "pm10"
      offset: 8
      len: 2
      decode: "uint16_be"

map:
  telemetry:
    "/air_quality":
      pm25: { field: "pm25", unit: "ug/m3" }
      pm10: { field: "pm10", unit: "ug/m3" }

2. Actuator Interface JSON

JSON files in interfaces/ define controllable outputs. When the node starts, all interface controls are merged with sensor capabilities and uploaded to the Server — making them available to the AI assistant and the command API.

File format

{
  "name": "differential_drive",
  "description": "Two-wheel differential drive controller",
  "communication": {
    "type": "uart",   // "uart" | "i2c" | "modbus" | "gpio"
    "bus": "uart0"    // matches uarts key in config.yaml
  },
  "controls": [
    {
      "name": "linear_velocity",
      "category": "actuator",
      "data_type": "float",
      "unit": "m/s",
      "range": [-1.0, 1.0],
      "command_topic": "/cmd_vel",
      "description": "Forward/backward speed"
    },
    {
      "name": "angular_velocity",
      "category": "actuator",
      "data_type": "float",
      "unit": "rad/s",
      "range": [-3.14159, 3.14159],
      "command_topic": "/cmd_vel",
      "description": "Rotation speed"
    }
  ]
}

controls[ ] fields

Field Required Description
nameYesActuator name — used in command API "actuator" field
categoryYesAlways "actuator"
data_typeYes"float" / "int" / "bool"
unitNoPhysical unit string (for AI context)
rangeRecommended[min, max] — Safety Gate enforces this range
command_topicNoLogical topic name for routing (informational)
descriptionRecommendedHuman-readable description shown to AI assistant

How AI uses actuator definitions

When a user types a natural language command (e.g. "move forward at 0.3 m/s") in the app, the AI assistant receives the full list of actuator capabilities including names, ranges, and units. It generates a structured command referencing the exact actuator name, which the Safety Gate then validates before execution.

3. Safety Rules YAML

The file at safety_rules.yaml defines the Edge Safety Gate. Rules are evaluated before every actuator command. Matching rules can block, clamp, or warn.

Format

rules:
  - name: "low_battery_stop"
    condition: "battery_level < 10"
    applies_to: "*"              # "*" = all actuators, or list a specific actuator name
    action: "block"              # "block" | "clamp" | "warn"
    message: "Battery critical — all actuator commands blocked"

  - name: "obstacle_stop"
    condition: "distance_cm < 20 AND linear_velocity > 0"
    applies_to: "linear_velocity"
    action: "block"
    message: "Obstacle too close — forward movement blocked"

  - name: "linear_velocity_clamp"
    condition: "true"            # always applies
    applies_to: "linear_velocity"
    action: "clamp"
    range: [-1.0, 1.0]          # values outside range are clamped, not blocked
    message: "linear_velocity clamped to [-1.0, 1.0] m/s"

Condition syntax

Conditions reference live status fields from the robot status map. Multiple conditions can be chained with AND:

condition: "battery_level < 10"
condition: "temperature_c > 75 AND speed > 0"
condition: "distance_cm < 20 AND linear_velocity > 0"
condition: "true"    # unconditional (always evaluates to match)

Supported operators: < > <= >= == !=

Action types

Action Behaviour
blockRejects the command entirely. Returns error to sender with the rule message.
clampAllows command but clips value to range: [min, max]. No error returned.
warnAllows command, logs a warning, and sends alert to Server.

Default safety_rules.yaml

rules:
  - name: "low_battery_stop"
    condition: "battery_level < 10"
    applies_to: "*"
    action: "block"
    message: "Battery critical (<10%) — all actuator commands blocked"

  - name: "low_battery_warn"
    condition: "battery_level < 20"
    applies_to: "*"
    action: "warn"
    message: "Battery low (<20%)"

  - name: "obstacle_stop"
    condition: "distance_cm < 20 AND linear_velocity > 0"
    applies_to: "linear_velocity"
    action: "block"
    message: "Obstacle detected within 20cm — forward motion blocked"

  - name: "linear_velocity_clamp"
    condition: "true"
    applies_to: "linear_velocity"
    action: "clamp"
    range: [-1.0, 1.0]
    message: "linear_velocity clamped to safe range"

  - name: "angular_velocity_clamp"
    condition: "true"
    applies_to: "angular_velocity"
    action: "clamp"
    range: [-3.14159, 3.14159]
    message: "angular_velocity clamped to safe range"
Dual-layer safety: The Safety Gate runs on the Node (edge). The AI assistant also performs its own safety evaluation on the Server before emitting commands. Both must pass for a command to reach the hardware.

4. Adding a New Sensor — Step by Step

  1. Create my-sensor.yaml in the protocols/ directory with the protocol definition.
  2. In config.yaml, add an entry under uarts: (or i2c:) referencing protocol: "my-sensor".
  3. Restart the Node: sudo systemctl restart rivo-node
  4. The Node logs should show protocol loaded: my-sensor.
  5. Telemetry data will appear under the configured topic in the Server and App.

5. Adding a New Actuator — Step by Step

  1. Create my-actuator.json in the interfaces/ directory of your installation.
  2. Add safety rules in safety_rules.yaml if needed.
  3. Restart the Node.
  4. The actuator name will appear in the AI assistant's capability context automatically.
  5. Send commands via the REST command API or directly via LiveKit DataChannel.
