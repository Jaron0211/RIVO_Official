Security & OTA

1. Security Architecture

Encryption model

  • Login: Credentials are RSA-OAEP encrypted with the server's ephemeral RSA public key before being sent. The session key negotiated during login derives the AES key for all subsequent requests.
  • API payloads: All protected request and response bodies are AES-256-GCM encrypted. The key is SHA-256(access_token). Plaintext is never transmitted after login.
  • JWT tokens: Access tokens expire in 30 minutes. Refresh tokens expire in 7 days. Sessions are stored server-side and can be revoked individually or all at once.
  • Command channel: The Node's WebSocket command channel uses the same AES-GCM encryption. Commands arriving over the LiveKit DataChannel are also validated by the Edge Safety Gate before execution.

Security best practices

  • JWT secrets: Use at least 32 random characters for access_secret and refresh_secret. Generate with: openssl rand -hex 32
  • HTTPS in production: Place Nginx or Caddy as a reverse proxy with TLS termination in front of :8080.
  • Network isolation: Only expose ports 8080, 7880, 7881 (TCP) and 7882 (UDP) to the internet. All internal services should be on a private network.
  • Node credentials: Use a dedicated low-privilege account for each Node. Do not use the master account credentials on edge devices.
  • Database: Never use the default DB_PASSWORD placeholder. Use a randomly generated password and store it only in the .env file.

2. Edge Safety Gate

Every actuator command received by the Node — whether from the REST command channel or the LiveKit DataChannel — passes through the Edge Safety Gate before any hardware is touched.

Gate evaluation order

  1. Receive command (actuator name + value)
  2. Check all rules in safety_rules.yaml that applies_to this actuator (or "*")
  3. Evaluate each rule's condition against the latest robot status snapshot
  4. On first matching block rule → reject command, return reason to sender
  5. Apply all matching clamp rules → adjust value within range
  6. Log matching warn rules and send alert to Server
  7. Execute the (possibly clamped) actuator write

See IO Definition → Safety Rules for the full rule format.

3. Offline Watchdog

The Node continuously monitors server connectivity. When connectivity is lost, the watchdog triggers the configured fallback policy to keep the robot in a safe state.

Config

safety:
  watchdog_interval: "5s"    # probe interval
  offline_threshold: 3        # consecutive probe failures before fallback
  fallback_policy: "stop"     # "stop" | "hold" | "none"

Fallback policies

Policy Behaviour on disconnect
stopZeroes all actuator outputs immediately. Safe default for mobile robots.
holdKeeps the last commanded value. Suitable for stationary equipment (e.g. conveyor speed).
noneNo automatic action. Commands simply stop arriving — hardware coasts.

When connectivity recovers, the Node resumes normal operation and logs a reconnection alert.

4. OTA Update — RIVO Server

The Server supports in-place binary updates without SSH access. Updates are pulled from GitHub Releases and applied by the rivo-updater companion process.

Architecture

RIVO Server (API)
    │
    ├── GET  /api/v1/system/versions   → list GitHub Releases (5-min cache)
    ├── POST /api/v1/system/update     → trigger update (master only)
    ├── GET  /api/v1/system/update/status → poll progress
    │
    ▼
rivo-updater (companion, port 8090)
    │
    ├── Downloads new binary from GitHub Release asset URL
    ├── Validates integrity
    ├── Writes to /opt/rivo-server/rivo-server (bind-mounted into container)
    ├── Runs: docker restart rivo-server
    └── Rolls back on failure

Update states

idle → downloading → applying → health_check → success
                                          └──→ rolled_back
                                          └──→ rollback_failed

Trigger an update (API)

# 1. List available versions
GET /api/v1/system/versions

# 2. Trigger update (requires master role)
POST /api/v1/system/update
# Encrypted body:
{ "version": "v1.3.0" }

# 3. Poll progress
GET /api/v1/system/update/status
# Returns: { "state": "applying", "target_version": "v1.3.0", ... }
Note: The rivo-updater binary itself is NOT updated via OTA — it is baked into its own Docker image. To update the updater, re-run the install.sh script on the host.

5. OTA Update — RIVO Node

RIVO Node supports remote binary update via commands delivered over the WebSocket command channel.

Command format

POST /api/v1/robots/{robotId}/commands
{
  "id": "ota-001",
  "type": "ota_update",
  "parameters": {
    "asset_url": "https://github.com/.../rivo-node-linux-arm64.zip",
    "version": "v1.3.0"
  }
}

What happens on the Node

  1. Node downloads the zip from asset_url (SSRF-validated, only HTTPS GitHub URLs allowed)
  2. Validates ZIP contents (path traversal and zip bomb protection)
  3. Extracts binary to a temp path
  4. Calls syscall.Exec to replace itself with the new binary
  5. New binary starts, re-registers with Server, resumes telemetry

6. Operations & Maintenance

Health check

curl http://localhost:8080/health
# {"status":"healthy","timestamp":"..."}

System info

GET /api/v1/system/info
# {"version":"v1.3.0","git_commit":"abc123","build_time":"...","uptime":3600}

Session management

POST /api/v1/auth/logout       # revoke current session
POST /api/v1/auth/logout-all   # revoke all sessions (e.g. after credential exposure)

Database backup

# PostgreSQL
pg_dump rivo > backup-$(date +%Y%m%d).sql

# SQLite (hot backup)
sqlite3 data/rivo.db "VACUUM INTO 'backup.db'"

Log management

Server logs are written to stdout by default (Docker captures them). Set logging.level: debug in config.yaml for detailed traces during development. Use docker compose logs --since 1h rivo-server to review recent logs.
