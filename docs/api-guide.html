API Usage Guide

All RIVO Server API communication uses end-to-end encryption. This page explains the encryption model and documents the key API endpoints.

1. End-to-End Encryption

RIVO uses a two-layer encryption model:

LayerApplies toAlgorithm
LoginPOST /api/v1/auth/login onlyRSA-OAEP (server's public key)
All protected endpointsEvery request/response body after loginAES-256-GCM, key = SHA-256(access_token)

Every protected request body must be wrapped as:

{ "encrypted_data": "<base64(AES-GCM-ciphertext)>" }

And every response is returned in the same wrapper. The AES key is derived at runtime by taking the SHA-256 hash of the raw JWT access token string.

2. Authentication Flow

Step 1 — Fetch RSA public key

GET /api/v1/auth/key

Response:
{
  "public_key": "-----BEGIN PUBLIC KEY-----\n..."
}

Step 2 — Generate session key & encrypt credentials

# Pseudocode
session_key = random_bytes(32)          # 32-byte random key
session_key_b64 = base64(session_key)

plaintext = JSON.stringify({
  "username": "admin@example.com",
  "password": "SecurePassword123!",
  "session_key": session_key_b64
})

encrypted = RSA_OAEP_encrypt(public_key, plaintext)

Step 3 — Login

POST /api/v1/auth/login
Content-Type: application/json

{ "encrypted_data": "<base64(RSA ciphertext)>" }

Response (decrypt with session_key):
{
  "access_token": "eyJ...",
  "refresh_token": "eyJ...",
  "expires_at": "2026-03-07T12:30:00Z"
}

Step 4 — Use access token

All subsequent requests:

Authorization: Bearer <access_token>
Content-Type: application/json

# Body: AES-GCM encrypt with key = SHA256(access_token)
{ "encrypted_data": "<base64(AES-GCM ciphertext)>" }

Token refresh

POST /api/v1/auth/refresh
Content-Type: application/json

{ "refresh_token": "eyJ..." }

Response (plaintext — refresh endpoint is NOT encrypted):
{
  "access_token": "eyJ...",
  "refresh_token": "eyJ...",
  "expires_at": "..."
}

3. Robot Management

Register a robot

RIVO Node auto-registers on startup. Manual registration (for testing):

POST /api/v1/robots
Authorization: Bearer <token>

# Encrypted body:
{
  "robot_id": "node-001",        # hardware-derived ID or custom
  "robot_type": "edge-node",
  "robot_name": "Warehouse Bot #1",
  "group_id": ""                  # optional group UUID
}

Response:
{ "robot_id": "<uuid>" }

List robots

GET /api/v1/robots
Authorization: Bearer <token>

Response:
{
  "robots": [
    {
      "id": "...",
      "robot_id": "node-001",
      "robot_name": "Warehouse Bot #1",
      "robot_type": "edge-node",
      "online": true,
      "battery_level": 85.0,
      "firmware_version": "1.3.0"
    }
  ],
  "count": 1
}

4. Status & Telemetry

Push status (Node → Server)

POST /api/v1/robots/{robotId}/status
Authorization: Bearer <token>

# Encrypted body:
{
  "status": {
    "online": true,
    "uptime": 3600.5,
    "status": "Operating",
    "battery_level": 85.0,
    "cpu_load": 0.45,
    "memory_usage": 4500000,
    "firmware_version": "1.3.0",
    "sensor_overview": [
      { "sensor_id": "PM2.5", "sensor_type": "Particulate Matter", "value": 35, "unit": "ug/m3" }
    ]
  }
}

Push telemetry (Node → Server)

POST /api/v1/robots/{robotId}/telemetry

# Encrypted body:
{
  "timestamp": "2026-03-07T10:00:00Z",
  "topics": {
    "/air_quality": {
      "pm25": 35.2,
      "pm10": 48.1
    },
    "/environment": {
      "temperature": 26.5,
      "humidity": 65.0
    }
  }
}

High-frequency WebSocket telemetry

When server.telemetry_ws: true is set in Node config, the Node uses a persistent WebSocket instead of HTTP POST for telemetry. This reduces connection overhead for high-frequency sensors (>2 Hz).

WebSocket: ws://<server>/api/v1/robots/{robotId}/telemetry/ws
Authorization: Bearer <token>

# Frames sent by Node (JSON):
{ "t": "/air_quality", "d": { "pm25": 35.2 }, "ts": 1709808000 }

5. Command & Control

WebSocket command channel (Node listens)

WebSocket: ws://<server>/api/v1/robots/{robotId}/commands
Authorization: Bearer <token>

# Server pushes commands to Node (AES-GCM encrypted):
{
  "id": "cmd-uuid",
  "type": "actuator_write",
  "parameters": {
    "linear_velocity": 0.3,
    "angular_velocity": 0.0
  },
  "timestamp": "2026-03-07T10:00:00Z"
}

# Node ACKs:
{ "command_id": "cmd-uuid", "status": "applied" }
# or rejected by safety gate:
{ "command_id": "cmd-uuid", "status": "blocked", "reason": "obstacle within 20cm" }

Publish command (REST → Node)

POST /api/v1/robots/{robotId}/commands
Authorization: Bearer <token>

# Encrypted body:
{
  "id": "cmd-uuid",
  "type": "actuator_write",
  "parameters": {
    "linear_velocity": 0.3
  }
}

LiveKit DataChannel realtime control

For latency-sensitive control (<20ms), the Android App sends control commands directly through the LiveKit room's DataChannel — bypassing the REST API entirely:

# DataChannel message format (JSON, sent by App):
{
  "type": "actuator_write",
  "params": {
    "linear_velocity": 0.5,
    "angular_velocity": -0.1
  },
  "seq": 42
}

# Node ACK (via DataChannel):
{
  "type": "ack",
  "seq": 42,
  "applied": true,
  "reason": ""
}

All DataChannel commands also pass through the Edge Safety Gate before reaching hardware.

6. Robot Capabilities (IO Query)

Query a robot's declared sensors and actuators — this is what the AI assistant uses:

GET /api/v1/robots/{robotId}/capabilities
Authorization: Bearer <token>

Response:
{
  "capabilities": [
    {
      "name": "pm25",
      "category": "sensor",
      "data_type": "float",
      "unit": "ug/m3",
      "range": [0, 999],
      "telemetry_topic": "/air_quality",
      "description": "PM2.5 particulate matter"
    },
    {
      "name": "linear_velocity",
      "category": "actuator",
      "data_type": "float",
      "unit": "m/s",
      "range": [-1.0, 1.0],
      "command_topic": "/cmd_vel",
      "description": "Forward/backward speed"
    }
  ]
}

7. Live Video Streaming

# Get LiveKit token for a robot's video room
GET /api/v1/robots/{robotId}/token
Authorization: Bearer <token>

Response:
{
  "token": "eyJ...",
  "host": "ws://<server-ip>:7880",
  "room": "<robotId>"
}

# Use token to connect to LiveKit room via LiveKit SDK
# The robot's camera track will be available as a video participant

8. Alerts

# Push alert from Node
POST /api/v1/robots/{robotId}/alert

# Encrypted body:
{
  "alert_type": "stream_error",
  "severity": "high",        # "low" | "medium" | "high" | "critical"
  "message": "Camera disconnected",
  "timestamp": "2026-03-07T10:00:00Z"
}

# List alerts
GET /api/v1/alerts?robot_id={id}&severity=high&limit=50

# Acknowledge
POST /api/v1/alerts/{alertId}/acknowledge

# Resolve
POST /api/v1/alerts/{alertId}/resolve
