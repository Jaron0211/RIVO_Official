Troubleshooting

Server Issues

"Invalid or expired token"

Cause: Access token has expired (default lifetime: 30 minutes).

Solution: Use the refresh token to get a new access token via POST /api/v1/auth/refresh. If the refresh token has also expired (7 days), the user must log in again.

"Connection refused" on port 8080

Cause: Server not running, wrong port, or firewall blocking.

Solution: Check docker compose ps or systemctl status rivo-server. Verify server.port in config.yaml. Check firewall rules with iptables -L or cloud security group settings.

"Database connection failed" on startup

Cause: PostgreSQL not reachable, wrong DSN, or missing password.

Solution: Verify database.dsn in config.yaml. Ensure DB_PASSWORD in .env matches the PostgreSQL user password. Test with: psql "your-dsn"

"Failed to decrypt payload" / response is garbled

Cause: AES key mismatch — the client is using a different access token than the one used to derive the encryption key.

Solution: Ensure the AES key is computed as SHA-256(raw_access_token_string) — the raw JWT string, not its decoded payload. Re-login to get a fresh token and recompute the key.

LiveKit video not working (black screen or timeout)

Cause: UDP port 7882 is blocked, or use_external_ip is not set correctly on NAT environments.

Solution:

  • Ensure UDP port 7882 is open inbound on your firewall / security group.
  • Set livekit.use_external_ip: true and optionally livekit.node_ip: "<public-ip>" in Server config.
  • Check Node logs for connected to livekit room — if absent, the Node failed to connect to the LiveKit signaling port (7880).
  • Never set livekit.force_tcp: true — it breaks WebRTC pairing with UDP clients.

Node Issues

Node fails to register / "authentication failed"

Cause: Wrong server.url, wrong credentials, or server not reachable from Node network.

Solution: Check config.yaml → server.url and auth.username / auth.password. Test connectivity: curl http://<server-ip>:8080/health from the Node.

Sensor data not appearing in app / telemetry missing

Cause: Protocol YAML not loaded, UART/I2C bus error, or wrong protocol ID referenced in config.

Solution:

  1. Check Node logs: journalctl -u rivo-node | grep protocol — look for protocol loaded: <id> or error messages.
  2. Verify that uarts.uart0.protocol in config.yaml exactly matches the id: field in the protocol YAML file.
  3. Check that the UART device exists: ls /dev/ttyS0 and the Node process has permission to access it.
  4. Enable debug logging: set logging.level: debug to see raw bytes received from the sensor.

"Protocol validation failed: unknown bus"

Cause: The bus: field in a protocol YAML contains an unsupported value.

Solution: Valid bus values: uart, i2c, spi, modbus, gpio, onewire, can. Check for typos.

Actuator command returns "blocked" / safety gate rejection

Cause: A safety rule condition was satisfied and the rule action is block.

Solution:

  • The command response includes a "reason" field with the rule message — read it to identify which rule fired.
  • Check the current robot status (battery, obstacle distance, etc.) that triggered the rule.
  • If the rule is too aggressive, edit safety_rules.yaml and restart the Node.
  • Common causes: battery below 10%, obstacle sensor reading <20 cm, actuator value outside declared range.

Node goes to safe-stop / all actuators zero unexpectedly

Cause: Offline watchdog triggered — the Node lost connectivity to the Server for more than offline_threshold consecutive probe intervals.

Solution:

  • Check network connectivity between Node and Server.
  • Review Node logs for offline threshold reached — triggering fallback: stop.
  • Increase safety.offline_threshold or safety.watchdog_interval if transient network glitches are causing false trips.
  • When connectivity restores, the Node automatically resumes normal operation.

Camera not streaming / "failed to start camera"

Cause: GStreamer not installed, camera device not accessible, or wrong device path.

Solution:

  1. Verify GStreamer is installed: gst-launch-1.0 --version
  2. Check device exists: ls /dev/video0
  3. Test camera directly: gst-launch-1.0 v4l2src device=/dev/video0 ! videoconvert ! autovideosink
  4. Check Node logs for the exact GStreamer pipeline being executed.
  5. Adjust cameras[0].width / height / framerate to values your camera supports.
  6. If using a CSI camera, set type: "csi" and provide a custom gst_pipeline.

I2C sensor not found / "identify check failed"

Cause: Wrong I2C address, wrong bus number, or sensor not connected.

# Scan I2C bus for connected devices
sudo i2cdetect -y 1

# Verify address matches protocol YAML
# Protocol YAML: address: "0x76"
# If sensor appears at 0x77 instead, update the YAML

WebSocket telemetry not connecting

Cause: server.telemetry_ws: true set but server WebSocket endpoint not reachable, or token expired during a long session.

Solution: The Node automatically falls back to HTTP POST telemetry if the WebSocket fails. Check logs for telemetry WS disconnected, falling back to HTTP. Verify port 8080 allows WebSocket upgrades through any reverse proxy.

General Diagnostic Commands

# Server logs (Docker)
docker compose logs --tail 100 rivo-server

# Node logs (systemd)
journalctl -u rivo-node -n 100 --no-pager

# Node status
systemctl status rivo-node

# Check if Node is registered on server
curl -H "Authorization: Bearer $TOKEN" \
  http://<server>:8080/api/v1/robots

# Check server system info
curl -H "Authorization: Bearer $TOKEN" \
  http://<server>:8080/api/v1/system/info
